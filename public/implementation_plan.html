Kế Hoạch Triển Khai - Curriculum Management Platform

Tài liệu này tóm tắt bài toán, phân tích các giải pháp kỹ thuật, đề xuất phân chia phạm vi (Scope MVP vs Long-term) và đưa ra các đề xuất cụ thể dựa trên ý kiến phản hồi của anh Khoa.

1. Tóm Tắt Bài Toán (Problem Summary)

Yap là ứng dụng học ngôn ngữ thông qua hội thoại giao tiếp (Speaking-first). Quy trình soạn thảo giáo trình và sản xuất audio hiện tại đang được thực hiện bán thủ công qua nhiều bước rời rạc:

  • Biên soạn kịch bản: Operator (chị Huyền) sử dụng prompt ChatGPT để sinh nội dung giáo trình theo 7 giai đoạn rồi điền vào các file Excel riêng lẻ cho từng ngôn ngữ.
  • Sản xuất âm thanh: Operator tải file Excel lên Google Colab (do anh Khoa thiết lập) để gọi API sinh âm thanh, sau đó tải về máy và upload thủ công lên Google Drive.
  • Đồng bộ ứng dụng: Lập trình viên di động (anh Khoa) phải kiểm tra chất lượng file và import dữ liệu thủ công vào database của ứng dụng.

2. Đề Xuất Thiết Kế & Phân Tích Trade-offs

Vấn Đề 1: Cấu Trúc Bài Học Động (Micro vs Non-Micro UI)

Bối cảnh: Tiếng Trung ở Level 1,2,3 kết hợp cả hai cấu trúc: Micro (3 bài đầu, có Tutor Script giải thích từng câu thoại) và cấu trúc cũ Non-Micro (các bài sau, chỉ có Dialogue hội thoại nhân vật). Các ngôn ngữ khác chỉ có Micro hoặc không có Micro.

  • Phương án đề xuất: Dynamic Tab Workspace (Chọn)

    Hệ thống phân biệt bài học qua trường lesson_type (micro/dialogue). Nếu là micro: Hiện 3 tab chỉnh sửa Vocabulary, Sentences và Tutor Script (_A*). Nếu là dialogue (bài cũ): Hiện tab Vocabulary và tab Dialogue Script (hội thoại qua lại giữa các nhân vật). Nếu ngôn ngữ không hỗ trợ Tutor Script: Ẩn hoàn toàn tab hội thoại.

    Trade-offs: Ưu tiên hàng đầu về mặt Simplicity cho Operator, tốc độ phát triển MVP nhanh, cấu trúc JSON xuất ra Mobile App cực kỳ ổn định.

  • Phương án 2: Notion-style Blocks (Không chọn)

    Cho phép Operator tùy ý tạo bất cứ loại thẻ nào. Phương án này quá phức tạp để hoàn thành kịp MVP và dễ gây lỗi dữ liệu khi xuất JSON.

Vấn Đề 2: Quản Lý Bản Dịch Đa Ngôn Ngữ Native (Localization Matrix)

  • Phương án A (Cột động hiển thị song song như Excel): Thuận tiện cho Operator review đối soát bản dịch và AI dễ sinh dữ liệu, nhưng tệ về thiết kế DB vì phải thay đổi schema cột vật lý mỗi khi thêm Native mới.
  • Phương án B (Translation Map - Quan hệ 1-nhiều): DB cực kỳ chuẩn hóa và mở rộng không giới hạn, nhưng con người và AI khó đối soát trực tiếp do dữ liệu bị tách dòng.
  • Đề xuất lựa chọn: Mô hình Hybrid (DB Chuẩn Hóa B - UI Hiện Thị Cột A)

    Database lưu trữ chuẩn hóa theo dạng 1-nhiều (Translations table). Tuy nhiên, API backend sẽ tự động JOIN và pivot dữ liệu để trả về dạng Cột động song song cho giao diện Web UI và AI Agent. Giúp đạt được cả hai mục đích: Con người dễ review, AI dễ viết, DB chuẩn hóa.

Vấn Đề 3: Cơ Chế QA Audio Tự Động & Sửa Lỗi

Tránh tình trạng AI tự động gọi sinh lại audio lỗi bị lặp vô hạn (infinite loop) gây chậm trễ toàn bộ tiến trình của batch:

  • Bước 1 (Batch-end Queue): Khi chạy batch sinh audio, câu thoại nào bị lỗi QA (ví dụ đọc nhầm tag) sẽ không thử sinh lại ngay lập tức mà bị đánh dấu lỗi và đẩy xuống cuối hàng đợi (queue at the end).
  • Bước 2 (Retry giới hạn): Sau khi xong toàn bộ batch, hệ thống thử chạy lại các file lỗi tối đa 2 lần ở cuối batch.
  • Bước 3 (Manual Fix Dashboard): Nếu vẫn lỗi, hệ thống hiển thị cảnh báo đỏ trên Dashboard kèm log chi tiết từ Whisper. Operator sửa text/tag lỗi trực tiếp trên giao diện rồi bấm Regenerate riêng câu đó.

3. Đề Xuất Phân Chi Phạm Vi (Scope MVP vs Long-term)

A. MVP Scope (Bàn giao 12:00 trưa Thứ Ba - 9/6)

Mục tiêu chính là xây dựng nền móng dữ liệu tập trung, cho phép CRUD nội dung bài học và từ vựng cơ bản thông qua giao diện dạng bảng, giúp anh Khoa lấy được JSON sạch cho Mobile App mà không cần file Excel.

        graph LR
          User[Marketing / Content Operator] -->|CRUD| WebPlatform[Curriculum Platform MVP]
          WebPlatform -->|Lưu trữ| SupabaseDB[(Supabase DB)]
          MobileApp[Mobile App API] -->|Query JSON sạch| SupabaseDB
  • Database Design (Supabase): Thiết kế và triển khai schema PostgreSQL hỗ trợ cả Micro và Dialogue cũ, đa native (EN, VI).
  • Giao Diện Grid Editor Cơ Bản (Minimal Airtable Clone):

    Soạn thảo, chỉnh sửa trực tiếp dạng bảng lưới đối với Từ vựng (Vocabulary) và Mẫu câu (Sentence) của từng bài học.

  • Xuất Bản Dữ Liệu (Export JSON API): API Endpoint xuất cấu trúc JSON sạch cho từng bài học phục vụ Mobile App.

B. Long-term Scope (Phát triển sau MVP)

  • Phase 2: Dialogue & Drill Editors

    Giao diện soạn thảo hội thoại Tutor theo dạng timeline; Công cụ tự động sinh các bài tập (Drill, Extra Drill) dựa trên ngân hàng từ vựng.

  • Phase 3: AI & Media Automation

    Tích hợp nút bấm sinh audio qua Gemini TTS; Dashboard hiển thị log lỗi của Whisper Audio QA cho phép nghe thử và sinh lại.

  • Phase 4: Localization Matrix & Sync

    Giao diện dịch thuật song song (side-by-side); cơ chế Webhooks đồng bộ trực tiếp lên Database Production của Mobile App.

4. Kế Hoạch Xác Minh (Verification Plan)

Automated Verification

  • Kiểm tra tính toàn vẹn của Database Schema bằng cách seed dữ liệu mẫu cho Level 1 của tiếng Trung (CN) và tiếng Nhật (JP).
  • Unit test cho API endpoint xuất JSON bài học, đảm bảo cấu trúc trả về khớp 100% với định dạng Mobile App.

Manual Verification

  • Người vận hành thử chỉnh sửa từ vựng và câu mẫu trên Grid Editor, kiểm tra xem dữ liệu trong Supabase/local file có cập nhật tức thời không.
  • Xuất file JSON bài học và gửi cho anh Khoa nạp vào ứng dụng để kiểm tra tính tương thích trên giao diện mobile.

5. Kế Hoạch Phối Hợp & Action Items Cho Anh Khoa (CTA)

Để triển khai suôn sẻ và đảm bảo tính tương thích cao nhất giữa Platform và Mobile App, dưới đây là các đầu việc và câu hỏi cần anh Khoa phối hợp xử lý / trả lời:

  • 1. Chốt cấu trúc Lesson Table trên Airtable/Notion:

    Anh Khoa đang thử nghiệm cấu trúc bảng bài học chuẩn hóa cho các ngôn ngữ khác nhau trên Airtable/Notion. Hành động (CTA): Khoa gửi lại bản thiết kế cấu trúc bảng hoàn chỉnh nhất cho Phúc để đồng bộ cấu trúc Schema trên Supabase. (Độ ưu tiên: Khẩn cấp).

  • 2. Duyệt thiết kế DB Schema (Supabase):

    File SQL schema.sql đã được tạo sẵn để triển khai trên Supabase. Hành động (CTA): Khoa xem qua định dạng bảng và mối quan hệ để xác nhận cấu trúc này đã tối ưu cho các truy vấn của Mobile App chưa.

  • 3. Đối soát định dạng JSON xuất bản (JSON Export Specification):

    API xuất JSON bài học đã có bản chạy thử nghiệm tại http://localhost:3000/api/lessons/JP_L101_MICRO/export. Hành động (CTA): Khoa tải thử file JSON này về và nạp thử vào app mobile (mock local) để kiểm tra xem có gặp lỗi parse dữ liệu hoặc lỗi hiển thị nào không.

  • 4. Thống nhất Cơ chế QA & Whisper local:

    Thống nhất quy ước đặt tên file audio (ví dụ segment_001.m4a khớp với Segment ID) và cách thức lưu trữ file log QA của Whisper. Hành động (CTA): Khoa xác nhận định dạng log lỗi mong muốn hiển thị trên Dashboard (VD: file JSON chứa status của từng segment) để Phúc thiết kế API endpoint lưu vết lỗi QA.
